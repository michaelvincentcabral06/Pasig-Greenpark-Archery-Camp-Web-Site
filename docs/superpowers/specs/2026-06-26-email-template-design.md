# Unified Branded Email Template — Design

## Summary

Make every transactional email share **one branded HTML shell** — a header band with the embedded logo, a consistent body card, and a consistent footer — so all six emails look uniform and carry the brand. Today four are hand-built HTML (slightly different from each other) and two (cancellation, reschedule) are plain text; none has the logo. Backend-only change in `backend/Code.gs` (the `send*_` functions), plus a small email-optimized logo asset. Needs a redeploy.

Owner-approved decisions: **embed the logo inline** (always shows, no "show images" needed); uniform frame with the title carrying a subtle **status tint** (confirmed=green, rescheduled=amber, cancelled=muted).

## The six emails (all refactored to the shared shell)

`sendReceipt_` (booking confirmed) · `sendPlanReceipt_` (pass confirmed) · `sendPlanSchedule_` (plan sessions scheduled/rescheduled/coach) · `sendPlanCancellation_` (pass cancelled) · `sendCancellation_` (booking cancelled — was plain text) · `sendReschedule_` (booking rescheduled — was plain text).

## New / changed pieces (backend, ES5 only)

1. **`assets/email-logo.png`** (NEW frontend asset) — a small email-optimized logo (~150px, transparent or on-brand, **< 30 KB**), generated by downscaling `assets/logo.png`. Committed + pushed to GitHub Pages so the backend can fetch it. Live at `https://michaelvincentcabral06.github.io/Pasig-Greenpark-Archery-Camp-Web-Site/assets/email-logo.png`.
2. **`logoBlob_()`** — returns the email logo as a Google `Blob` named `logo` for inline embedding:
   - Try `CacheService.getScriptCache()` for a cached base64 (key `emailLogoB64`, ~6h TTL); on miss, `UrlFetchApp.fetch(EMAIL_LOGO_URL)` → base64, cache it.
   - Returns `Utilities.newBlob(Utilities.base64Decode(b64), 'image/png', 'logo')`.
   - **Null-safe:** any failure (fetch error, asset missing) → returns `null`. The email must still send without the logo (header falls back to text only). Never throws into the send path.
3. **`emailShell_(title, accentColor, innerHtml, hasLogo)`** — returns the full branded HTML:
   - **Header band** (`background:#244232`): if `hasLogo`, `<img src="cid:logo" width="40" height="40" …>` + the business name in light text; else the business name text only.
   - **Body card** (`background:#fffdf6`, `max-width:520px`, centered): an `<h2>` title tinted with `accentColor`, then `innerHtml` (the email-specific content).
   - **Footer**: "Where" (`RANGE_ADDRESS` · `CONTACT_NUMBER`), the `RESCHEDULE_NOTE`, and `BUSINESS_NAME`.
   - Email-safe: tables/inline styles, `font-family:Arial,Helvetica,sans-serif`, no external CSS. Reuse `escapeHtml_`.
4. **`sendBranded_(o)`** — the single send path. `o: { to, subject, plainText, title, accent, innerHtml }`:
   - `var blob = logoBlob_();` build `emailShell_(o.title, o.accent, o.innerHtml, !!blob)`.
   - `MailApp.sendEmail({ to:o.to, subject:o.subject, body:o.plainText, htmlBody:html, name:BUSINESS_NAME, inlineImages: blob ? { logo: blob } : undefined })`.
   - Always include a readable **plain-text `body`** fallback (each send function provides one).
5. **Refactor the six `send*_`** — each builds its own `innerHtml` (the details table / message), a `plainText` fallback, picks its `title` + `accent`, and calls `sendBranded_`. Remove their bespoke `<div>…</div>` wrappers and `MailApp.sendEmail` calls. Keep their existing inputs/`o` params and the data they show.

## Status tints (accent per email)

- Confirmed (receipt, pass receipt): green `#3c6b48`.
- Scheduled (plan schedule): green `#3c6b48`.
- Rescheduled: amber `#8a6a1f`.
- Cancelled (booking + pass): muted red `#b4512f`.

(The frame/header/footer are identical; only the title text + tint differ.)

## Constraints

- **Backend ES5 only** (var/function); keep `backend/Code.gs` ≡ `Code.LATEST.gs` and add `Code.v21.gs`; bump the version marker to `db-v21` (+ `brandedEmail:true`).
- **No `index.html` change** beyond adding the `assets/email-logo.png` file (the mirror rule still holds — the new asset doesn't touch the HTML files).
- **Email size:** the embedded logo must be the small `email-logo.png` (< 30 KB), NOT the 1 MB `logo.png`.
- **Graceful degradation:** a missing/unfetchable logo never blocks or errors an email — it just sends without the image.
- **No secret/PII concerns** — these are existing customer emails; only their presentation changes.

## Deploy order (owner)

1. Push the repo (puts `assets/email-logo.png` live on Pages) — assistant does this.
2. Owner pastes the updated `backend/Code.gs`, Saves, and deploys a new version (db-v21). No Script Property changes.
   (If the backend deploys before the asset is live, emails simply send without the logo until the asset is up — no breakage.)

## Verification

- **Build-time (assistant):** render `emailShell_` sample output (a representative email of each type, with a placeholder logo box) in headless Chromium and screenshot it to confirm the HTML renders as a clean, uniform, branded layout (header/body/footer correct, title tints right, table aligned). Confirm ES5 (no `=>`/`const`/`let`/template-literals in new code) and `.gs` copies identical.
- **Post-deploy (owner):** confirm `?action=version` → `db-v21`; then trigger one of each email to themselves (book a session → receipt; reschedule → reschedule email; cancel → cancellation) and confirm they arrive **uniform, with the logo visible**.
